# 组织树与空间作用域设计

日期：2026-07-23  
状态：待实现评审

## 背景与决策

主场景：市/县相对省是**同一客户内的行政分支**（展示上要隔离，运营上仍是一家），不是独立客户。

**结论：一个 Tenant（空间）承载整棵行政树；下级知识库共享只用 OrgUnit + `share_with_descendants`。不为上下级共享去拆空间，也不走 Organization。**

| 概念 | 实体 | 职责 |
|------|------|------|
| 空间 | `Tenant` | 强隔离：配额、RBAC、配置、账单 |
| 组织树 | `OrgUnit` | 同一空间内的省→市→县；控制资源可见范围 |
| 共享给下级 | `share_with_descendants` | 祖先 KB 对子孙**只读** |
| 共享空间 | `Organization` + `kb_shares` | **仅**跨 Tenant 协作，不模拟行政上下级 |

一句话：**空间按客户/独立运营单元切；组织树按行政层级切；向下共享用树，跨客户协作用共享空间。**

## 边界划分

1. 客户 / 独立运营单元 → 一个 `Tenant`（例如「某省某局」）。
2. 省 → 市 → 县 → 同一 Tenant 内的 `OrgUnit` 树。
3. 上级知识库给下级用 → `org_unit_id` + `share_with_descendants=true`（子孙可读，写不向下传）。
4. 跨局 / 跨客户协作 → `Organization` + `kb_shares`（非日常上下级路径）。
5. `OrgUnitWorkspace`（平台根懒开通业务空间）只服务「平台根 → 开通业务 Tenant」，**不为每个市县自动开 Tenant**。

## 成员归属

约束（相对当前多组织能力的收紧）：

- 同一 Tenant 内，每个用户 **最多属于一个 OrgUnit**（禁止兼任）。
- 调岗 = **换绑**到另一个 OrgUnit，不是叠加成员关系。
- 当前组织 = 唯一所属 OrgUnit；不再用 `is_primary` 表达「多组织选主」。
- 无所属 OrgUnit 时，退回租户级遗留行为（兼容旧数据）。

实现要求：

- 服务层在创建/调整 `OrgUnitMember` 时拒绝第二份归属。
- 调岗 API：原子地解除旧绑定并写入新绑定。
- 若表结构仍保留 `is_primary`，产品语义不再依赖；后续可清理字段（非本设计阻塞项）。

空间成员（`TenantMember`）与组织归属正交：

- `TenantMember.role`：owner / admin / contributor / viewer。
- `OrgUnitMember`：决定树内可见性作用域。

## 知识库归属与可见性

| 动作 | 规则 |
|------|------|
| 新建 | 绑定到当前（唯一）OrgUnit；默认 `share_with_descendants=false` |
| 共享给下级 | 子孙 OrgUnit **可读**；不可改、不可删 |
| 本级写 | 空间 RBAC + Contributor「只能改自己的」 |
| 祖先共享库 | 出现在下级列表；用来源标记与自建库区分 |
| `org_unit_id` 为空 | 租户级遗留库：按空间 RBAC 可见；**新库不再默认走这条** |

Agent 与知识库同一套 OrgUnit 绑定与列表过滤原则。

明确不做：

- 共享给下级做成向上汇聚或兄弟节点互看。
- 为「看不到」去复制库或复制空间。

## 配额、管理入口、产品文案

### 配额与配置

- 配额、模型配置、账单、空间级 RBAC 全部挂在 **Tenant**。
- OrgUnit **不**各自开配额。
- 若某下级需要独立配额 / 独立 Owner / 可整体迁出 → 升格为独立 Tenant（例外路径，非默认）。

### 管理入口

| 入口 | 职责 |
|------|------|
| 组织（`/org-units`） | 树结构、单组织挂人、调岗换绑 |
| 成员（`/members`） | 空间成员与角色 |
| 知识库 / Agent 列表 | 按当前 OrgUnit 过滤；祖先共享只读 + 来源标记 |
| 设置中的深链 | 可保留，一级能力已外提 |

可见范围：

- 空间 Owner / 具备整树权限的角色：可管理整树。
- 普通成员：本级自建 + 祖先共享只读（及产品已定义的邀请子树规则）。

### 文案边界（避免两套「共享」混淆）

| 场景 | 用语 | 机制 |
|------|------|------|
| 省库给市县、市库给县 | **共享给下级** | OrgUnit + `share_with_descendants` |
| 两个客户空间协作 | **共享到共享空间** | Organization + `kb_shares` |
| 兄弟 OrgUnit 互看 | 默认不做 | 若硬性需要，走显式共享空间，不放开树横向 |

## 与现有代码的对齐

已有能力可直接承载本设计：

- `KnowledgeBase.org_unit_id` / `share_with_descendants`
- OrgUnit 树 `path` / `depth` 与 `CanRead` / `CanWrite` 语义
- 前端作用域标签、`ResourceOriginBadge`、组织/成员一级导航

需要收紧或补齐：

- **单 OrgUnit 归属**强制（API + 调岗换绑；必要时数据迁移清理一人多组织）
- 新产品文案区分「共享给下级」与「共享到共享空间」
- 确认新库创建路径始终写入当前 `org_unit_id`，避免再产生无组织遗留库（管理员显式租户级库若需要，另开入口）

## 非目标（本阶段）

- 按 OrgUnit 独立配额或自动为市县开通 Tenant
- 兄弟节点横向自动可见
- 用 Organization 模拟行政上下级
- 跨空间「共享给下级」的等价体验封装

## 成功标准

1. 同一客户下省市区县用户同空间登录，通过组织切换/归属看到正确 KB/Agent 集合。
2. 上级勾选「共享给下级」后，子孙只读可见；不能编辑/删除该库。
3. 无法将同一用户同时挂到两个 OrgUnit；调岗后可见集合随新组织变化。
4. 日常上下级共享不出现「必须先建共享空间再拉 Tenant」的路径。
5. 跨客户协作仍只通过 Organization，与树内共享文案可区分。
